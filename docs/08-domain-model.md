# 08. 업무 도메인 모델 (퀵/화물 배차 중개)

> 실제 채널톡 상담 데이터(2026-06-02)에서 도출한 업무 구조. 본 프로젝트의 **핵심 데이터 모델**이며, AI 로봇이 다룰 대상이다.

## 업무 한 줄 정의
논스톱서비스는 **거래처(고객사)가 채널톡으로 보낸 퀵/화물 운송 요청을 받아, 기사에게 배차하고, 배차정보·운임을 안내하는 중개(주선) 업무**를 한다.

## 등장 인물(역할)

| 역할 | 실제 예 | 시스템상 |
|------|---------|----------|
| 거래처(고객사) | 하림, 리씽크 | `clients` |
| 거래처 담당자 | 조용호, 김병준, 오세란 | `client_contacts` (신규) |
| 상담원(논스톱 직원) | 정태신, 오현미, 방미라 | `agents` |
| 기사 | 김광석, 이성태, 김장하 | `drivers` (신규) |

## 핵심 업무 흐름

```
[거래처 담당자]  ──퀵요청(채널톡)──▶  [상담원]
                                        │ 1) 오더 정보 추출/접수
                                        │ 2) 기사 배차(매칭)
                                        ▼
                              [기사] ◀──배차정보 전달──
                                        │ 3) 운임 안내(거래처)
                                        │ 4) 운행 추적(지도/위치)
                                        │ 5) 재배차(기사 이탈 시)
                                        ▼
                                    운행 완료 → 정산/세금계산서
```

## 실제 메시지에서 본 두 가지 핵심 양식

### A. 오더(접수) 요청 — 거래처 → 논스톱
> "퀵요청드립니다. 경유 3곳 / 즉시입차가능. 하차지 매장안에 넣어주세요!"
> 출발지: 일승푸드 제2공장 / 경기도 용인시 기흥구 지삼로 116 / 010-3209-4004
> 도착지(경유1) 11박스: 대전 동구 계족로 429 장수통닭 / 010-9401-3754
> 도착지(경유2) 1박스: … / 도착지(경유3) 2박스: …
> 물품: 종이박스 총 14개(닭 부분육) / 차량: 다마스

**→ 추출해야 할 필드(오더):**
| 필드 | 예시 | 비고 |
|------|------|------|
| 요청유형 | 즉시 / 시간지정 | "즉시입차가능", "11시" |
| 출발지 | 상호·주소·연락처 | |
| 경유/도착지[] | 상호·주소·연락처·수량 | **다중**(경유 N곳) |
| 물품 | 종이박스 14개(닭 부분육) | 종류·수량 |
| 차량종류 | 다마스 / 오토바이 / 라보 / 1톤 | |
| 특이사항 | "하차지 매장안에 넣어주세요" | 자유 텍스트 |
| 경유 순서 | 무관 / 지정 | "코스는 편하게" |

### B. 배차정보 — 논스톱 → 거래처(배차 완료 안내)
> 배차정보
> 일산설문동▶구로개봉동
> 이성태 / 010-6435-3742 / 서울91자2960
> 운임 40,000원(공급가)

**→ 생성해야 할 필드(배차):**
| 필드 | 예시 |
|------|------|
| 구간 | 출발지▶도착지(인원/수량) |
| 기사명 | 이성태 |
| 기사 연락처 | 010-6435-3742 |
| 차량번호 | 서울91자2960 |
| 차종 | 오토 / 다마스 |
| 운임 | 40,000원(공급가) |

### 그 외 본 패턴
- **재배차**: "길석우 기사님 빠지셔서 재배차 중입니다." → 배차 상태 전이
- **운행 추적**: 지도 이미지 + "[기사▶출] 10.317Km" (기사~출발지 거리)
- **확인 문의**: "경유 순서는 상관없는 걸까요?" → "상관없습니다, 코스는 편하게"

## AI 로봇이 할 일 (재정의)

기존 설계(FAQ 답변)보다 핵심은 **오더 처리 자동화**다.

| 단계 | AI 작업 | 출력 |
|------|---------|------|
| 1. 접수 파싱 | 자유형식 요청 메시지 → **구조화 오더**(structured extraction) | orders 레코드 |
| 2. 유형 분류 | 메시지가 접수/배차/운임/변경/클레임 중 무엇인지 | category |
| 3. 응답 생성 | 배차정보 양식, 운임 안내, 확인 답변 초안 | 답변 초안 |
| 4. FAQ/정책 | 경유 규칙·운임 기준 등 반복 질문 답변 | RAG 답변 |

> ①번(접수 파싱)이 가장 가치가 크다. 수기로 입력하던 오더를 메시지에서 자동 구조화 → 데이터로 축적되면 배차·운임·정산까지 연결된다.

## 데이터 모델 추가 (기존 스키마에 더할 테이블)

```
clients(거래처) 1──< client_contacts(담당자)
clients 1──< orders(오더/접수)
orders 1──< order_stops(경유/도착지)   # 다중 경유 지원
orders 1──< dispatches(배차)
dispatches >── drivers(기사)
conversations ──(연결)── orders          # 어떤 상담에서 나온 오더인지
```

- **orders**: 요청유형, 출발지(주소/연락처/상호), 물품, 차량종류, 특이사항, 상태(접수/배차중/배차완료/운행/완료/취소)
- **order_stops**: order_id, 순번, 유형(경유/도착), 상호, 주소, 연락처, 수량, 경유순서무관여부
- **dispatches**: order_id, driver_id, 차량번호, 차종, 운임(공급가), 상태(배차/재배차/운행/완료), 배차시각
- **drivers**: 이름, 연락처, 차량번호, 차종

> 상세 컬럼/SQL은 [02 DB 스키마](02-database-schema.md)에 반영 예정(다음 작업).
